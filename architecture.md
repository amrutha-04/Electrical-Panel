# Microgrid Panel Designer Architecture


## 1. Product Overview

Microgrid Panel Designer is a desktop application (Python + pywebview + HTML/CSS/JS) that allows a user to:
- enter microgrid design inputs,
- generate SLD and GA diagrams,
- compute BOM and engineering metrics,
- export:
  - Technical Report PDF,
  - GA PDF,
  - BOM Excel.

## 3. Runtime Layers

### Layer A: Desktop Host
Files:
- `main.py`
- `app.py` (launcher alias)
- `run_app.py` (launcher alias)

Responsibilities:
- create window via pywebview,
- set window dimensions/position,
- load `ui/index.html`,
- start GUI backend.

Important runtime note:
- `main.py` sets `QT_API=pyside6` by default to ensure Qt backend consistency.

### Layer B: Bridge/API Layer
Files:
- `api/bridge.py`

Responsibilities:
- receives payloads from frontend JS,
- manages app state (`theme`, `last_payload`, optional MCCB DB),
- orchestrates calculations and SVG generation,
- handles export requests and save dialogs,
- enforces export diagrams in light style for documents.

Primary bridge methods:
- `get_state()`
- `set_theme(theme)`
- `generate(payload)`
- `export_pdf(payload)`
- `export_ga_pdf(payload)`
- `export_excel(payload)`

### Layer C: Core Adapter Layer
Files:
- `core/bom.py`
- `core/ga.py`
- `core/sld.py`
- `core/utils.py`
- `core/constants.py`

Responsibilities:
- thin compatibility and orchestration adapters,
- stable import surface between bridge and domain modules,
- export composition (especially PDF/Excel in `core/bom.py`).

### Layer D: Engineering Domain Layer
Files under `src/`:

- `src/constants.py`
- `src/utils.py`

- `src/sld/`
  - `calculations.py`
  - `components.py`
  - `generator.py`

- `src/ga/`
  - `dimensions.py`
  - `generator.py`
  - `styles.py`

- `src/bom/`
  - `generator.py`
  - `exports.py`

Responsibilities:
- electrical sizing calculations,
- GA geometry and panel layout,
- SLD symbol and topology rendering,
- BOM item derivation and export support.

### Layer E: Frontend Layer
Files under `ui/`:
- `ui/index.html`
- `ui/style.css`
- `ui/app.js`

Responsibilities:
- page layout and component structure,
- theming and responsive styling,
- event handling and API integration,
- rendering SVG previews and metrics,
- fullscreen preview interactions.

## 4. End-to-End Data Flow

### Generate flow
1. User edits inputs in `ui/index.html` controls.
2. `ui/app.js` builds payload and calls `window.pywebview.api.generate(payload)`.
3. `api/bridge.py` normalizes input and computes design through `core/*` -> `src/*`.
4. Bridge returns summary + SVGs + BOM data.
5. `ui/app.js` updates:
   - SLD/GA previews,
   - warning/status card,
   - metrics summary.

### Export flow
1. User clicks export button in `ui/app.js`.
2. Bridge export method recomputes/uses payload.
3. `core/bom.py` builds PDF/Excel bytes.
4. Bridge opens save dialog and writes file.

## 5. Theme Model

- UI theme is toggleable (dark/light) in frontend.
- Export documents are generated in light style for print readability.
- SLD/GA color usage was refactored to reduce hardcoded dark colors and use theme-aware values.

## 6. Important Recent Changes (From Start to Current State)

This is the practical history of major architecture-impacting changes:

1. App entry and UI modernization
- consolidated runtime around `main.py` + pywebview + `ui/`.
- retained `app.py` and `run_app.py` as compatibility launchers.

2. Bridge-centered orchestration
- moved generation/export coordination through `api/bridge.py`.
- added consistent theme state handling and export behavior.

3. Output panel UX refactor
- action buttons moved beside "Generated Output" heading.
- output order changed to diagrams first, warnings section, metrics later.

4. Manual generate behavior
- returned to explicit Generate button workflow (disabled auto-regenerate on every input mutation).

5. Fullscreen and zoom/pan
- per-diagram fullscreen buttons added.
- blurred overlay added.
- zoom + panning interactions added in fullscreen mode.

6. GA preview behavior split
- GA web preview hides side spec table for better usage of available space.
- GA export file keeps full table content.

7. Theme hardening in diagrams
- GA/SLD residual dark hardcoded colors reduced and made theme-aware.
- SLD Solar and MGC symbol colors updated to avoid dark artifacts in light mode.

8. PDF/asset packaging fixes
- `main.spec` updated to include logo asset (`Kirloskar Oil Engine Logo.png`).
- PDF footer logo handling in `core/bom.py` updated with robust path resolution for source and frozen (EXE) runtime.

9. Packaging backend stabilization
- runtime standardized toward PySide6/Qt path.
- packaging conflict with mixed Qt bindings was addressed by avoiding multiple active Qt bindings in build path.

10. Repository cleanup
- reduced generated/unnecessary files and old docs.
- kept architecture in `architecture.md` and high-level usage in `README.md`.

## 7. Source Tree Details

### `api/`
- `bridge.py`: JS/Python API boundary and export orchestration.

### `core/`
- `bom.py`: PDF/Excel construction, footer/logo behavior, schedule table composition.
- `ga.py`, `sld.py`: adapters/wrappers to domain generators.

### `src/`
- `src/sld/*`: SLD math + drawing primitives + generation.
- `src/ga/*`: panel geometry + styling + GA SVG generation.
- `src/bom/*`: BOM item modeling and export helpers.

### `ui/`
- `index.html`: structure and control IDs.
- `style.css`: themes, layout, responsive behavior, scrolling, fullscreen visual styling.
- `app.js`: state + event wiring + bridge calls + preview rendering.

### Utility / data files
- `BusBar_dimensions.xlsx`: lookup data for sizing.
- `Pole_modification.xlsx`: related engineering data sheet.

### Legacy/auxiliary files
- `Electricpanel.py`: legacy monolithic script retained for reference.
- `app.py.backup`: backup file (not primary runtime path).

## 9. Known Constraints

1. Frontend IDs are tightly coupled to `ui/app.js` handlers.
2. pywebview runtime depends on valid GUI backend availability.
3. PyInstaller builds can fail if conflicting Qt bindings are present simultaneously.
4. Any new static asset used by UI or exports must be included in packaging data.

## 10. Dependencies and Why They Are Used

This section documents the practical package set used by this project.

### 10.1 Direct Project Dependencies (requirements.txt)

1. `pandas`
- Used for tabular BOM handling and Excel export workflows.

2. `svgwrite`
- Used to construct SLD/GA diagrams as SVG programmatically.

3. `openpyxl`
- Used to write BOM output to `.xlsx` and handle workbook-level operations.

4. `reportlab`
- Used to generate PDF files (technical report and GA PDF), including footer/header text and logo drawing.

5. `svglib`
- Used to convert SVG content into ReportLab drawing objects for embedding diagrams into PDF pages.

6. `Pillow`
- Image support used by export/graphics stack (including PDF image handling paths).

7. `pywebview`
- Desktop host bridge between Python backend and HTML/CSS/JS frontend.

8. `pyinstaller`
- Used to package app into Windows executable (`dist/main.exe`).

9. `pywin32`
- Windows-specific integration support used by desktop/runtime behavior and related dependencies.

### 10.2 Runtime GUI Backend Dependencies

These are critical for pywebview desktop execution in current setup:

1. `PySide6`
- Qt GUI binding used as the desktop rendering backend.

2. `QtPy`
- Qt abstraction layer used by pywebview's Qt platform module.

3. `shiboken6`
- Required runtime component for `PySide6`.

4. `bottle` and `proxy_tools`
- pywebview support packages used by its internal HTTP/bridge mechanisms.

### 10.3 Packaging Support Dependencies

1. `pyinstaller-hooks-contrib`
- Extra PyInstaller hooks to correctly collect packages/resources.

2. `pywin32-ctypes`, `pefile`, `altgraph`
- Used by PyInstaller build process on Windows.

### 10.4 Transitive Dependencies

Many additional packages seen in `pip list` are transitive dependencies (for example `numpy`, `lxml`, `tinycss2`, `requests`, `urllib3`, `tzdata`, etc.).
They are not directly coded against in this app but are required by the direct dependencies above.

### 10.5 Dependency Source of Truth

- Primary tracked list: `requirements.txt`
- Active environment snapshot: `venv` (`pip list --format=freeze`)

If package behavior changes, update both `requirements.txt` and this section.

## 11. How to Extend Safely

### Add new input field
1. Add control in `ui/index.html`.
2. Wire state/event in `ui/app.js`.
3. Normalize in `api/bridge.py`.
4. Consume in `src/*` calculation/generation.

### Add new export type
1. Add button in `ui/index.html`.
2. Add click handler in `ui/app.js`.
3. Add bridge method in `api/bridge.py`.
4. Add export builder in `core/bom.py` or dedicated module.

### Add new asset (logo/icon/background)
1. Place asset in stable location.
2. Reference from UI/export code.
3. Include asset in `main.spec` `datas`.

## 12. Quick Onboarding Map

If you are new and want to debug quickly:

1. Start app/window issue -> `main.py`
2. API payload or export issue -> `api/bridge.py`
3. PDF/Excel issue -> `core/bom.py`
4. SLD content issue -> `src/sld/generator.py` and `src/sld/components.py`
5. GA layout issue -> `src/ga/generator.py`
6. UI layout issue -> `ui/index.html` + `ui/style.css`
7. UI interactions -> `ui/app.js`
